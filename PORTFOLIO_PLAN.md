3. 철학적 문제 구조화와 SSAFY 기술 이해를 연결하는 PM
4. AI, 데이터, 웹 기술을 이해하고 서비스 의사결정으로 연결하는 기획자
5. 문제정의, 사용자 여정, 기술 선택을 함께 설계하는 AI 서비스 PM

## 2. 최종 추천 포지셔닝 1개와 이유

추천 포지셔닝:

> AI/SW 기반 문제정의형 서비스 기획자 / PM

이유:
- 기존 `AGENTS.md`에서 이미 핵심 방향으로 고정한 표현이다.
- "비전공 철학 전공"을 약점이 아니라 문제정의와 구조화 역량으로 연결하기 좋다.
- SSAFY 경험을 단순 개발 교육이 아니라 AI/SW 기술 이해와 협업 기반 실행 경험으로 설명할 수 있다.
- DailyLog, Loggy, Tamna-AI, Promtree처럼 서로 다른 프로젝트를 "문제정의 → 서비스 구조화 → 기술 선택"이라는 하나의 축으로 묶을 수 있다.

## 3. 메인 화면 섹션 순서

현재 구조를 유지한다.

1. Hero
   - 5초 안에 정체성 전달
   - 포지셔닝, 핵심 증거, 대표 프로젝트 CTA 제공
2. About
   - 철학 전공과 SSAFY 기술 이해의 연결 설명
   - 사용자 이해, 문제 구조화, 논리적 설계 역량 강조
3. Projects
   - 제출용 핵심 프로젝트 3개를 먼저 노출
   - 나머지는 Archive 또는 Supporting Projects로 분리
4. Competencies
   - 프로젝트에서 반복적으로 드러난 역량을 정리
   - 문제정의, 기술 이해, AI 서비스 설계, 협업/문서화 중심
5. Experience
   - SSAFY, 현장 경험, 철학 전공을 보조 근거로 제시
   - 프로젝트보다 앞서지 않게 유지

## 5. Skills/Competencies 분류 기준

권장 분류:

1. Problem Framing
   - 문제를 표면 요구가 아니라 반복되는 병목과 사용자 행동 구조로 재정의
2. Service & UX Structuring
   - 사용자 여정, 화면 흐름, 입력 부담, 결정 흐름을 서비스 구조로 설계
3. AI/SW Technical Understanding
   - AI, 데이터, 웹 기술을 서비스 가치와 실행 가능성 관점에서 해석
4. Collaboration & Documentation
   - 팀 방향 정렬, 발표자료, 문서화, 의사결정 기록을 산출물로 관리

주의:
- 기술 스택 나열만으로 끝내지 않는다.
- 각 역량은 반드시 최소 1개 프로젝트 증거와 연결한다.

## 6. Projects 카드 공통 템플릿

각 카드에 포함할 정보:

1. 프로젝트명
2. 한 줄 문제정의
3. 내 역할
4. 핵심 기술 또는 구조
5. 결과/증거
6. 면접 질문 유도 포인트

카드 문장 구조:

```text
[프로젝트명]
어떤 문제를 발견했는가:
내가 맡은 역할:
어떤 기술/구조를 선택했는가:
결과 또는 배운 점:
```

## 7. 상세 페이지 공통 템플릿

상세 페이지 권장 순서:

1. Project Overview
   - 기간, 팀, 역할, 상태, 핵심 키워드
2. Problem Definition
   - 기존 문제, 사용자 불편, 비즈니스/운영 병목
3. My Role
   - PM/기획/프론트엔드/발표/문서화 등 실제 역할
4. Technical Choices
   - 기술을 왜 선택했는지
   - 단순 스택 나열 금지
5. Solution Process
   - 문제 해결 과정, 의사결정, trade-off
6. Result / Learning
   - 수상, 완성도, 발표, 배운 점
   - 수치가 없으면 TODO로 표시
7. Interview Hooks
   - 면접에서 물어볼 만한 질문 3개

## 8. 제출용 핵심 프로젝트 3개 후보와 선정 이유

추천 1순위 조합:

1. DailyLog
   - 이유: SSAFY 특화 프로젝트 우수상 수상 사실이 기존 데이터에 있다.
   - 강점: AI 질문 흐름, 기록 지속 문제, 추천 구조, PM 역할이 가장 선명하다.
   - 면접 질문: 왜 기록 문제를 의지 부족이 아니라 구조 문제로 봤는가?

2. Loggy
   - 이유: 협업 의사결정 문제를 Git 구조로 재해석한 사례다.
   - 강점: 철학적 문제 구조화와 SW 협업 도구 이해가 연결된다.
   - 면접 질문: 왜 일반 메신저가 아니라 Issue/PR/Merge 구조를 차용했는가?

3. Promtree
   - 이유: 현재 프로젝트 데이터에 문제정의, 시장성, AI 노하우 실행 구조, 수익모델, 경쟁사 분석 기여가 비교적 풍부하게 존재한다.
   - 강점: AI 서비스 PM, 시장성 검토, 양면시장, 크레딧 모델 등 취업용 질문을 만들기 좋다.
   - 면접 질문: 프롬프트 마켓이 아니라 "실행 가능한 AI 노하우 상품"으로 정의한 이유는 무엇인가?

대안 조합:

1. DailyLog
2. Loggy
3. Tamna-AI

Tamna-AI 선택 이유:
- 재생에너지, 분산 데이터센터, 입지 분석이라는 도메인 확장성이 있다.
- 데이터 기반 의사결정과 사회 문제의 서비스화 관점이 드러난다.

Tamna-AI 보류 이유:
- 제출용 핵심 3개로 쓰려면 실제 산출물, 사용 데이터, 분석 방식, 결과 검증 자료가 더 필요하다.
- 부족한 내용은 TODO로 보완해야 한다.

S.A.L.E.S / CRM 보류:
- `src/content/projects.js`에 TODO 주석으로 "자료 보완 필요"가 명시되어 있다.
- 현재 단계에서는 핵심 3개보다 보조 프로젝트로 두는 것이 안전하다.

Campus Chronicle 보류:
- 성장 데이터 기반 금융 서비스라는 아이디어는 좋지만, 현재 핵심 포지셔닝인 AI/SW 기반 문제정의형 PM과의 직접 연결은 DailyLog/Loggy/Promtree보다 약하다.
